权限修饰符和代码块

发表于 2023-05-15 分类于 技术
本文字数： 842 阅读时长 ≈ 1 分钟

#权限修饰符
定义：是用来控制一个成员变量能够被访问的范围的

修饰符 同一个类中 同一个包中的其他类 不同包下的子类 不同包下的无关类
private √ × × ×
空着不写 √ √ × ×
protected √ √ √ ×
public √ √ √ √

注：实际开发中，一般只用private和public

  • 成员变量私有
  • 方法公开

特例：如果方法中的代码是抽取其他方法中的共性代码，这个方法一般也私有

代码块

局部代码块

  • 特点：提前结束变量的生命周期，即使释放内存
    public class Text {
  public static void main(String[] args) {  
      {
          int a = 10;
          System.out.println(a);
      }// 局部代码块
  }
}

构造代码块

  • 特点：抽取构造方法中的重复代码
  • 格式：写在成员位置
  • 执行时机：再创建本类对象的时候会先执行构造代码块再执行构造方法
public class Text {
    private String name;
    private int age;
    {
    System.out.println("看看我执行了吗")
    }
    public Text() {
    }
    public Text(String name, int age) {
        this.name = name;
        this.age = age;	
    }
}

静态代码块

  • 特点：通过static关键字修饰，随着类的加载而加载，并且自动触发、只执行一次
  • 格式：static{}
  • 使用场景：在类加载的时候，做一些数据初始化的时候使用

抽象方法和抽象类

抽象方法

  • 定义：将共性的行为（方法）抽取到父类之后。由于每一个子类执行的内容是不一样，所以，在父类中不能确定具体的方法体。该方法就可以定义为抽象方法
  • 格式：public abstract 返回值类型 方法名(参数列表);

抽象类

  • 定义：如果一个类中存在抽象方法，那么该类就必须声明为抽象类
  • 格式：public abtract class 类名{}

注意事项

  • 抽象类不能实例化
  • 抽象类中不一定有抽象方法，有抽象方法的类一定是抽象类
  • 可以有构造方法
  • 抽象类的子类：要么重写抽象类中的所有抽象方法；要么是抽象类

包和final

发表于 2023-05-12 分类于 技术
本文字数： 449 阅读时长 ≈ 1 分钟

什么是包？

包就是文件夹。用来管理不同功能的java类，方便后期代码维护
命名规则：公司域名 + 包的作用，需要全部英文小写，见名知意。
如：com.companyname.domain

什么是全类名？

包名 + 类名，组成全类名（全限定名）。如：com.companyname.domain.Text

导包？

避免每次都写全类名，诞生了”导包“

  • 使用同一个包中的类时，不需要导包
  • 使用java.lang包中的类时，不需要导包
  • 其他情况都需要导包
  • 如果同时使用两个包中的同类名，需要用全类名

final修饰符？

修饰方法时：表示该方法是最终方法，不能被重写
修饰类时：表示该类是最终类，不能被继承
修饰变量时：叫做变量，只能被赋值一次

常量的意义：在实际开发中，常量一般作为系统的配置信息，方便维护，提高可读性
常量的命名规则：

  1. 单个单词：全部大写
  2. 多个单词：全部大写，单词之间用下划线隔开

小细节：

  • final修饰的变量是基本类型：那么变量存储的地址值不能发生改变
  • final修饰的变量是引用类型：那么变量存储的地址值不能发生改变，对象内部可以改变

最近突然勤快了起来，连续写了几篇“小作文”。就觉得自己上天了，身体轻飘飘的——哇！我这么快的产出率，岂不是很快就要诞生出许多“大作”了，就算是迅哥儿当年也就是这个量了哈哈。不过细细想来，公众号、贴吧、知乎、CSDN、便笺……如此多的“作品”我还并没有发布在同一个平台，不仅杂乱无章，而且实在是不利于管理。于是，一个荒诞却又挺符合逻辑的想法诞生了——为什么我不创建一个自己的博客网站呢？在我的网站，我想写什么就写什么，我想什么时候发布就什么时候发布。审核？老子自己就是审核！其实主要是为了装杯杯哈哈哈虽然最近接触了JAVA，不过开发网站——不对口啊！于是打开我的百度“如何搭建一个博客网站”“啪”的就跳出一大堆的教程，真是让人眼花缭乱。果然顿哥真理——如果你觉得我看的很远，那是因为我站在巨人的肩膀上。

研究了一会儿，我便开始想要口吐芬芳了“服务器——要钱；域名——要钱”尤其是服务器，价格简直就是出淤泥而不染，濯清涟而不妖，中通外直，不蔓不枝，香远益清，亭亭净植，可远观而不可亵玩焉。某一时刻，我突然诞生了一个荒谬的想法——互联网开源时代，大家都在讲“开源精神”，为什么服务器要掏钱，这很不开源！可是，难道我就这么放弃了吗？是的，我关掉了标签页，然后打开了B站。哈哈哈哈，摆烂人生不需要解释。但“博客风波”真的就这么完结撒花了吗？当然不是，要不然也就不会有这么一篇“作品”了。“各位看官你细听分说~”说起来也巧，就在我沉浸于优质舞蹈区（也就是我的动态）欣赏那曼妙绝伦的舞姿间隙，我看到了一个视频，其标题是“手把手教你上线网站，不用服务器！小白可懂”哦，原来我还关注过这种up主啊！真少见。在万般不情奈之下，我抱着“实在不想看”的心态打开了视频，可能当时我的心思并不在上面，视频的内容不出我所料“毫无卵用”。其实也不能这么说，因为我在旁边的推荐视频中看到了一个单子“hexo”。欸！感觉在哪里见过，但是再多的细节又没有一丝的印象。既来之则安之，看看吧。

于是，我发现了新天地——白嫖github。哈哈哈，是谁这么天才想到这种方法，牛逼！于是我在百度上又找到了文字版的hexo+github搭建网站的教程。开始了我的建站之路，“漫漫长征路，过程很‘艰辛’”，不过聪明如我，最终开始完成了网站的搭建，虽然很简陋就是了。主要还是因为我没啥要求——能用就行。如你所见，确实只能用“能用“来形容，什么美轮美奂酷炫派、什么可爱动人二次元，跟我的一比，就像是停在共享单车旁边的劳斯莱斯哈哈哈

抛开这些花里胡哨，我还是挺满足的。毕竟拥有一个自己的网站就一件多么爽的一件事情“这就是自由！“我再也不用忍受传东西先审核的傻狗机制，我再也不用忍受各种敏感字的傻狗规则，我再也不用忍受写了东西但却找不到的傻狗服务器。从这一刻，我自由了，就像是一个从笼子里挣脱出来的一个大口呼吸着外面新鲜空气的活生生的人，就像一个摆脱了母鹰的阴影在更大的天空下自由翱翔的逐风的金鸟。虽然还在空天之下但至少，我在向前走了……
